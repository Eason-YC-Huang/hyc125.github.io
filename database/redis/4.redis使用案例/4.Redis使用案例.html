分布式锁

获取锁：SET ${key} ${value} NX PX ${timeout}命令。
释放锁：
-- KEYS[1] = ${key} ARGV[1] = ${value}
if redis.call("get",KEYS[1]) == ARGV[1] then
    return redis.call("del",KEYS[1])
else
    return 0
end
关键点解析：
  • 设置timeout，防止获取到锁的客户端崩溃了，没有释放锁，导致锁泄露
  • 设置value，防止客户端操作超时，锁被自动释放，别的客户端获取到锁，前客户端操作完成把后客户端的锁释放掉
  • 释放锁使用lua脚本：释放锁包含三步操作：'GET'、判断和'DEL'，用Lua脚本来实现能保证这三步的原子性。
尚未解决的问题：
假如Redis节点宕机了，那么所有客户端就都无法获得锁了，服务变得不可用。为了提高可用性，我们可以给这个Redis节点挂一个Slave，当Master节点不可用的时候，系统自动切到Slave上（failover）。但由于Redis的主从复制（replication）是异步的，这可能导致在failover过程中丧失锁的安全性。考虑下面的执行序列：
  1. 客户端1从Master获取了锁。
  2. Master宕机了，存储锁的key还没有来得及同步到Slave上。
  3. Slave升级为Master。
  4. 客户端2从新的Master获取到了对应同一个资源的锁。
于是，客户端1和客户端2同时持有了同一个资源的锁。锁的安全性被打破。

分布式计数信号量

计数信号量可以限制一项资源最多能够同时被多少个用户访问。(限流)
使用Redis来实现超时限制特性通常有两种方法：
  • 使用EXPIRE命令
  • 使用有序集合
这里我们使用有序集合实现计数信号量
每个尝试获取信号量的用户分配一个唯一标识符，然后将该标识符用作有序集合的成员，成员对应的分值是用户尝试获取信号量时的时间戳。
用户检查自己的标识符在有序集合中的排名，如果排名低于可获取的信号量总数，那么表示进程成功地取得了信号量。否则获取信号量失败，它必须从有序集合中移除自己的标识符。
PS：为了处理过期的信号量，添加标识符到有序集合之前，先清理有序集合中时间戳大于超时数值的标识符。
.
private String semaphoreKey = "semaphore";
private long timeout = 1000 * 10;
private int size = 2;


public boolean acquireSemaphore(String identify) {
  long now = System.currentTimeMillis();
  // 清除过期的记录
  this.jedis.zremrangeByScore(semaphoreKey, "-inf", String.valueOf(now - timeout));
    // 添加到有序集合
  this.jedis.zadd(semaphoreKey, now, identify);
    // 获取排名
  Long rank = this.jedis.zrank(semaphoreKey, identify);
  return rank < size;
}

public Long releaseSemaphore(String identify) {
  return this.jedis.zrem(semaphoreKey, identify);
}
