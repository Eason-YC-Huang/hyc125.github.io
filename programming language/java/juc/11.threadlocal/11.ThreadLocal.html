基本使用

ThreadLocal：线程局部变量。
// 创建ThreadLocal
private ThreadLocal<String> myThreadLocal = new ThreadLocal<String>();
// 设值
threadLocal.set("A thread local value");
// 获取值
String threadLocalValue = myThreadLocal.get();
// 删除值
threadLocal.remove();
// 初始化时设值
// 1.重写initialValue()方法
// 2.使用ThreadLocal的静态工厂方法，该方法需要提供一个Supplier接口的实现（推荐）
private ThreadLocal<String> uuid = ThreadLocal.withInitial(() -> UUID.randomUUID().toString());

原理

Thread类有一个ThreadLocalMap类型的实例变量。
ThreadLocalMap是定义在ThreadLocal类的一个静态内部类
ThreadLocal的 get()、set() 方法其实就是获取当前线程的ThreadLocalMap对象，然后调用该对象对应的 get()、set() 方法。
例如下面的 set 方法：
public void set(T value) {
  Thread t = Thread.currentThread();
  // 获取当前线程的ThreadLocalMap对象
  ThreadLocalMap map = getMap(t);
  if (map != null)
    map.set(this, value);
  else
    createMap(t, value);
}

内存泄漏问题

为什么ThreadLocalMap的Entry是通过继承WeakReference来实现的？
static class Entry extends WeakReference<ThreadLocal<?>> {
    /** The value associated with this ThreadLocal. */
    Object value;

    Entry(ThreadLocal<?> k, Object v) {
        super(k);
        value = v;
    }
}
当你使用一个ThreadLocal对象时，至少会有两个地方引用了这个ThreadLocal对象的。
public static void main(String[] args) {
    ThreadLocalCounter counter = new ThreadLocalCounter();
    // T1 
    new Thread(counter::get).start();
}

public static class ThreadLocalCounter {
    private ThreadLocal<Integer> cnt = ThreadLocal.withInitial(() -> 0);

    public int get() {
        return cnt.get();
    }
}
例如上面的代码，假设ThreadLocalCounter和Entry都强引用了ThreadLocal<Integer> cnt，当ThreadLocalCounter被回收后，由于Entry对ThreadLocal<Integer> cnt的强引用，会导致cnt无法被回收。
假设Entry对ThreadLocal<Integer> cnt是弱引用。当Entry调用get方法获取ThreadLocal<Integer> 时得到null，说明这个Entry已经没有用了，可以清理掉。
ThreadLocalMap是用线性探测法解决的hash冲突的。当它发现某个位置已经占用后，会顺便检查该位置的Entry是否有效，如果无效就删掉。
